# learning-game — 単元特化ゲーミフィケーション学習アプリ生成スキル

指定された小・中学校の1単元について、**その単元で身につけるべき全ての知識・技能を段階的に練習でき、
ゲーミフィケーションで学習意欲が持続する**学習ゲームアプリを構築（新規生成 or 既存アプリ拡張）する。

進め方の大原則:

1. **情報が揃うまで実装しない。** Phase 0 のチェックリストに不足があれば `AskUserQuestion` で
   何度でも質問し、揃った場合のみ Phase 1 へ進む。
2. **ループ機能で完了まで自走する。** 実装開始後はタスクリストを作り、/loop（動的モード。
   なければ自前で ScheduleWakeup / 継続イテレーション）を使って「実装 → 検証 → 修正」を
   タスク完了まで繰り返す。中途半端な状態でターンを終えない。
3. **検証してから納品する。** 型チェック・ビルド・問題生成のプロパティテスト・ブラウザでの
   スモークテストが全て通ってからコミット・push・ドラフトPR作成。

---

## Phase 0 — 情報収集（不足があれば実装せず質問を繰り返す）

以下のチェックリストを埋める。ユーザーの依頼文・セッション内のリポジトリから読み取れるものは
自分で埋め、**判断がつかない項目だけ** AskUserQuestion でまとめて質問する（1回で最大4問。
足りなければ回答を受けて再度質問する）。

| # | 項目 | 備考 |
|---|---|---|
| 1 | 学年・教科・単元名 | 例: 小4 算数「わり算の筆算」／中2 理科「電流」 |
| 2 | 対象アプリ | セッション内に既存の学習アプリがある → それを拡張。ない → 新規生成（`references/app-scaffold.md`）。どちらか判断がつかなければ質問 |
| 3 | 教科書会社（任意） | 東書/啓林館/光村など。用語・指導順・仮商の立て方等が変わる。不明なら「一般的な構成」で進めてよい（その旨を伝える） |
| 4 | 参考実装 | 既定は `onokomachi/syo4-syousu`（または `onokomachi/wari-hissann3`）。セッションに追加できればコードを直接コピーして流用。アクセス不可なら `references/gamification.md` の仕様から再実装 |
| 5 | 【国語のみ】本文 | 教科書本文は著作権保護されているため**ユーザーに本文の貼り付けを依頼する**（必須）。青空文庫等パブリックドメイン作品なら取得を試みてよい。本文が得られない場合は、本文非依存の問題（漢字・語彙・文法）に絞る旨を確認する |
| 6 | 開発ブランチ・push先 | 環境から指定があればそれに従う。なければ確認 |

全項目が埋まったら「これで実装を開始します」と要点をまとめてから Phase 1 へ。

## Phase 1 — リサーチ（単元の知識・技能と頻出問題の洗い出し）

1. **学習指導要領ベースの整理**: この単元の「知識・技能」「思考・判断・表現」を箇条書きにする。
   WebSearch が使えれば学習指導要領解説・単元テストの大問構成を調べる。使えなければ
   自身の知識と `references/question-design.md` の教科別カタログで構成する。
2. **頻出問題タイプの列挙**: 問題集・単元テストで定番の問題タイプを網羅的にリストアップし、
   「基礎 → 標準 → つまずきポイント → 応用」の**段階的なレベル**に並べる。
   つまずきポイント（系統的な誤答パターン）は必ず独立したレベル/モジュールにする。
3. **回答方法の選定**: 問題タイプごとに最適な入力方式を決める
   （`references/question-design.md` の「回答UIカタログ」参照）。
   タイマーは付けない。誤答には理由と次の一手を示す精緻化フィードバックを設計する。

リサーチ結果を「モジュール構成表（モジュール × レベル × 回答UI × skillId）」として
ユーザーに見える形で提示してから実装に入る。

## Phase 2 — 設計

- **ハブ型構成**: ホーム「〈単元名〉ランド」にモジュールカードを並べる。5〜8モジュールが目安。
  うち1つは必ず「エラーハンター（まちがい探し）」、1つは「本番テストモード」。
- **skillId 設計**: `<モジュールprefix>-<レベル>` 形式。progressStore の
  `skillToModuleId()` に prefix を登録する。
- **本番テストモード**: 実際の単元テストを模した 表=知識・技能100点（5点×20問目安）／
  裏=思考・判断・表現50点（10点×5問目安）。一発正解のみ得点。答案詳細を学習のきろくに保存。
- **ゲーミフィケーション**: `references/gamification.md` の全機能（バッジ・テーマ解放・
  目標リング・連続記録・熟達バー・おまかせ適応難易度・ふくしゅう提案・効果音・称賛・
  学習のきろく）を単元名・モジュール名に合わせて最適化して全部入れる。

## Phase 3 — 実装（ループで完了まで）

1. TaskCreate でタスクを分割（基盤移植 → ハブ/きろく/設定 → 各問題モジュール → 本番テスト → 検証/PR）。
2. 参考リポジトリを clone できる場合は、そのまま流用できるファイル
   （store / lib（sound, praise, themeUnlock, useAdaptive, badges 骨格）/ ui（GoalRing,
   primitives, レイン背景）/ shared（Keypad, AnswerEntry, AppShell, AdaptiveBar）/
   index.css のテーマトークン / index.html の FOUC 防止スクリプト）をコピーして単元用に改名・最適化する。
3. **問題ジェネレーターは純粋関数**として `src/lib/problems.ts` に集約する
   （UIから分離してテスト可能にする。乱数生成で問題数は実質無限、リトライ上限つき）。
4. 各モジュールは「レベル選択（LevelCard＋熟達バー＋今日のクリア数）＋おまかせモード」の
   共通パターン。1問の進行は Round コンポーネントに分離し、本番テストから再利用できる形にする。
5. 新規アプリの場合は `references/app-scaffold.md` の手順で土台を作ってから 2〜4 を行う。

## Phase 4 — 検証（全部通るまで修正ループ）

1. `tsc --noEmit` と `vite build` が成功する。
2. **プロパティテスト**: 全ジェネレーター×全レベルを各1万回生成し、恒等式
   （例: わられる数＝わる数×商＋あまり）・値域・レベルごとの出題形状・正答インデックスの
   整合・誤答例が本当に誤りであること等を検証するスクリプトを書いて実行する（tsx で実行）。
3. **Playwright スモークテスト**: dev サーバーを起動し、ハブ→各モジュール1問→本番テスト→
   学習のきろく→設定（テーマ切替）を巡回してコンソールエラーが無いこと・スクリーンショットで
   表示崩れが無いことを確認する。
4. 失敗があれば修正して 1 に戻る（これがループの本体）。

## Phase 5 — 納品

1. 指定ブランチにコミット（日本語の説明的メッセージ）・push し、ドラフトPRを作成する。
2. ユーザーへの最終報告: 追加したモジュール/レベル一覧、ゲーミフィケーション機能一覧、
   検証結果、PRリンク、次の一歩（マージ・PR見守り・調整の提案）。
3. タスクが全て完了したらループを終了する。

## 品質基準（達成必須）

- 単元の知識・技能を**漏れなく**カバー（Phase 1 の構成表と実装の突き合わせを最後に行う）
- 全テキストは対象学年が読める表記（学年相当より上の漢字はひらがな、分かち書き気味に）
- 誤答時: 正誤だけでなく「なぜ・次にどうするか」を示す。人格でなくプロセスを称賛する
- ミスしても罰しない（連続ノーミスが切れるだけ）。タイマー・不快なブザー音は使わない
- 全テーマ（light/和/マトリックス/深海/極光/宇宙神）で文字が読めることをスクリーンショットで確認
- localStorage キーは単元アプリごとに一意（例: `<app>_progress_v1` / `<app>_settings_v1`）
